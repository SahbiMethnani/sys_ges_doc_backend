Besoins Fonctionnels

Gestion des Tickets

  • BF-01
    Création et gestion complète des tickets
    Créer, modifier, assigner, commenter et archiver des tickets de type bug, feature, support, refactor ou documentation. Gestion des priorités P1–P4 et des statuts (open → doing → testing → accomplished/failed).
    HAUTE
  • BF-02
    Lifecycle unifié des stories et bugs
    Gérer le cycle de vie standardisé : TODO → DOING → TO_TEST → TESTING → ACCOMPLISHED ou FAILED. Seul le développeur peut modifier les statuts dev ; seul le testeur peut modifier les statuts test. Blocage possible avec commentaire obligatoire.
    HAUTE
  • BF-03
    Suivi du temps par ticket
    Enregistrer les heures passées sur chaque ticket dans dev_time_trackings, avec vue analytique par développeur et par application.
    MOYENNE

Navigation & MindMap

  • BF-04
    Navigation MindMap interactive
    Visualiser l'arborescence de chaque application sous forme de carte mentale (MindElixir). Ajouter, modifier, supprimer des nœuds avec synchronisation temps réel vers la base de données. Menu contextuel : Documentation, Créer ticket, Demander à l'IA.
    HAUTE
  • BF-05
    Navigation synchronisée MindMap ↔ KoolReport
    Synchroniser la sélection d'un nœud dans la MindMap avec l'affichage des tables de gestion dans KoolReport et vice-versa.
    MOYENNE

Agents IA

  • BF-06
    Pipeline IA Plan → Validation → Exécution
    Détecter les tickets labellisés (ai-claude, ai-opencode, ai-gemini) en moins de 60s. Générer un plan en lecture seule (phase 1), attendre validation humaine, puis exécuter les modifications (phase 2). Deux budgets distincts : $2/10min (plan) et $5/30min (exécution).
    HAUTE
  • BF-07
    Routing intelligent des LLMs
    Calculer un score de complexité via le Code Context Hub et router automatiquement vers le modèle le plus approprié : Ollama local (gratuit) pour les tâches simples, Claude Opus/GLM pour les tâches complexes, selon la stratégie de coût (abonnements illimités > LLMs locaux > API pay-per-use).
    HAUTE
  • BF-08
    Mémoire vectorielle (RAG)
    Indexer documentation, commits, tickets résolus et schémas DB dans Qdrant. Enrichir les prompts des agents IA avec le contexte pertinent (top_k=8, cosine similarity ≥ 0.25). Collections : orchestrator_docs, wikijs_rag, db_schemas, decisions, tickets.
    HAUTE
  • BF-09
    Dashboard de monitoring IA
    Afficher en temps réel l'état des agents, l'historique des exécutions, les stats par agent et les métriques quotidiennes sur 30 jours. Chat RAG intégré pour interroger la base de connaissances.
    MOYENNE

Intégration GitLab

  • BF-10
    Webhooks GitLab bidirectionnels
    Traiter les événements Push (lier commits aux tickets via #id), Merge Request (résoudre ticket automatiquement au merge), Pipeline (créer ticket si pipeline échoue), Issue (synchroniser issues GitLab).
    HAUTE
  • BF-11
    Synchronisation automatique des commits
    Indexer les commits humains dans Qdrant (orchestrator_decisions) toutes les heures via cron, pour enrichir la mémoire IA des décisions techniques prises par l'équipe.
    MOYENNE

Multi-Tenant & Documentation

  • BF-12
    Architecture Multi-Tenant Master/Clients
    L'instance MASTER collecte les tickets de toutes les instances clientes via API REST sécurisée (API key). Synchronisation bidirectionnelle : le client envoie les tickets vers le master, le master envoie les mises à jour vers les clients.
    HAUTE
  • BF-13
    Documentation auto-générée par IA
    Générer et mettre à jour automatiquement la documentation de chaque page à partir du code source. Détecter les pages sans documentation et suggérer un contenu. Éditeur Quill intégré pour édition manuelle.
    MOYENNE
  • BF-14
    Tests automatisés et prévention des régressions
    Enregistrer des tests (unitaire, intégration, e2e, regression, performance, security) avec frameworks PHPUnit, Jest, Jasmine, Playwright, Cypress. Bloquer les déploiements si un test critique échoue.
    HAUTE
  • BF-15
    Tracking des fonctionnalités par page
    Surveiller la présence des fonctionnalités critiques (boutons, menus, formulaires, API) via CSS selectors (Playwright), comparaison screenshots ou appels API. Créer automatiquement un ticket si une fonctionnalité disparaît.
    HAUTE
  • BF-16
    Planning développeurs (personnel + équipe)
    Vue calendrier personnelle (créer/modifier événements, déclarer congés, lier aux tickets). Vue équipe (multi-développeurs, détection conflits, charge de travail, vue Gantt, drag & drop). Recherche automatique du développeur disponible selon compétences requises.
    MOYENNE

Besoins Non-Fonctionnels

Performance

  • BNF-01
    Délai de détection des tickets IA
    L'orchestrateur doit détecter un ticket labellisé en moins de 60 secondes (intervalle de polling configurable). Temps de réponse du Chat RAG : ~3 secondes (Ollama GPU).
    HAUTE
  • BNF-02
    Limites de budget IA
    Budget quotidien maximal : $50/jour avec alerte à $20. Budget par ticket : $2/10min (phase plan), $5/30min (phase exécution). Maximum 2 retries par ticket, skip après 5 échecs consécutifs.
    HAUTE
  • BNF-03
    Index sémantique régénéré en temps raisonnable
    Le Code Context Hub indexe 4740 entrées et doit être régénéré toutes les heures via cron sans impacter les performances de l'API. Temps de recherche vectorielle < 500ms.
    MOYENNE

Sécurité

  • BNF-04
    Authentification JWT pour l'API
    Toutes les requêtes vers /Api/sales/dev/* doivent être authentifiées via JWT. Les instances clientes utilisent des API keys uniques pour la synchronisation multi-tenant.
    HAUTE
  • BNF-05
    Contrôle des permissions des agents IA
    Les agents IA s'exécutent sous l'utilisateur système agent-ia (uid 1000) en mode bypassPermissions. Chaque agent possède un compte GitLab distinct (@agent-maestro, @agent-coder…) pour traçabilité complète des actions.
    HAUTE
  • BNF-06
    Soft delete obligatoire
    Aucune suppression physique : toutes les tables utilisent un champ disable pour les suppressions logiques, garantissant l'auditabilité et la restauration des données.
    HAUTE
  • BNF-07
    Validation du webhook GitLab
    Vérifier le token X-GitLab-Token sur chaque webhook entrant. Rejeter avec HTTP 401 toute requête non authentifiée.
    HAUTE

Disponibilité & Maintenabilité

  • BNF-08
    Services supervisés par systemd
    Tous les services critiques (orchestrateur, code context hub, Angular, Nginx, PHP-FPM) sont gérés par systemd avec redémarrage automatique. Monitoring via Grafana + Loki + Promtail.
    HAUTE
  • BNF-09
    Traçabilité complète des actions IA
    Stocker dans SQLite (orchestrator.db) l'historique de toutes les exécutions (issue_id, phase, status, agent, started_at, durée, coût). Accessible via API /api/history et interface dashboard.
    MOYENNE
  • BNF-10
    Documentation API complète
    Chaque endpoint API expose des exemples de requête/réponse JSON documentés. L'architecture de fichiers suit une convention stricte (/Api/sales/dev/[module].php).
    MOYENNE

Scalabilité & Extensibilité

  • BNF-11
    Architecture multi-tenant extensible
    Le système doit supporter l'ajout d'instances clientes sans modification du master. L'API de synchronisation est conçue pour être générique et réutilisable par tous les modules (TASKS, NETWORK, ORES…).
    HAUTE
  • BNF-12
    Catalogue LLM extensible
    L'orchestrateur doit pouvoir intégrer de nouveaux modèles LLM (abonnements, locaux, API) sans refonte architecturale. La table ia_providers / ia_models (à créer) centralisera les configurations.
    BASSE
  • BNF-13
    Concurrence des agents configurable
    Claude Code : séquentiel (1 ticket/cycle). OpenCode/GLM-4-Plus : jusqu'à 20 tâches concurrentes. La concurrence doit être configurable par agent sans redémarrage du service.
    MOYENNE

Compatibilité & Internationalisation

  • BNF-14
    Support multilingue de la MindMap
    Les textes de la carte mentale doivent supporter l'internationalisation via la table dev_map_i18n (FR, EN, NL…).
    BASSE
  • BNF-15
    Encodage UTF-8 et charset unifié
    Toutes les tables MySQL doivent utiliser utf8mb4 avec index FULLTEXT pour la recherche de texte. Les nouvelles tables (dev_tests, dev_features_tracking) appliquent déjà ce standard.
    MOYENNE
